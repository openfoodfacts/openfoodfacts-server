#This file uses the following structure:
#- A compliance set defines the rules for product cross sections. This can be allergen rules (gluten free), origin rules (sustainable palm free), religious rules (halal), etc. Each set can have multiple compliancy levels. There will always be a default compliancy level. Each compliancy set is preceded by a comment ("compliance set X section") for easy human reading. A compliancy set is defined by a tag, such as *gluten free compliant*  . A human understandable tag is preferred.
#- A compliancy set should comprise of two or more levels. These levels are ordered, so that one level can be more compliant than the other. This order is indicated by an positive number (a higher number implies better compliance). Thus a compliancy level consists of a compliancy set tag and a number (*vegan compliant:en:level:en:3*).
#- Each compliancy as a default compliancy level. The default simplifies the definition of a compliancy set. A product has a default compliancy level, unless one of the other levels is triggered. The default can be any of the levels of a compliancy set. The default is defined by for instance: *nova level compliant:en:default:en:yes* is added on a separate line.
#- In order to understand each level better, an explanatory tag can be added, such as *pescatarianism compliant:en:label:en:product contains certification label*. This could be used as key for translation purposes.
#- A compliancy level tag is followed by one or more taxonomy keys. These taxonomy keys act as triggers for a level. A taxonomy consists of the name of the relevant taxonomy and the key within that taxonomy. For instance *additive:en:E962*. For human readability keys from the same taxonomy should be kept together and the keys should be alphabetically ordered.
#
#
#Each product can be matched against one or more compliancy sets. The information of each product is analysed and matched against the taxonomies. The product keys can then be matched against the keys listed in each compliancy set. This matching should include all the parents that are available in the taxonomies. If the keys are matched the corresponding level is triggered.
#
#It might occur that triggers contradict each other. This can be due to contradictory information on the product itself or issues with the determination of the product keys. These could be flagged as quality issues. For instance a product with beef and a vegetarian label is clearly wrong.
#
#These contradictions might imply that one level takes precedence over another level. The principle is that the higher level is right. For instance a product might contain ingredients that trigger nova level 1, but just one nova level 4 ingredient might triggers level 4 and that will be assigned to the entire product.
#
#There is also the possibility that a parent and child key contradict each other. In that case the child is right, as it contains more specific information. For instance a product might have the ingredient *en:fat*, which origin is animal or vegetable, and the ingredient *en:palm*, which is  from vegetable origin. Then the *en:palm* key takes prevalence.
#
#
#In order to maintain the integrity of the compliant file, several tests can be performed:
#- One can define the compliancy set internal integrity in multiple ways.
#    - No double levels
#    - Taxonomy keys are unique over the entire compliancy set
#    - Each compliancy set contains one and the same compliancy set tag
#    - (the existence of a parent-child relation could be added, so that it can be tested)
#- The taxonomy names should be one of the available enumerated taxonomies (ingredient, additive, category, label, mineral, enzyme, other).
#- Each key should exist in the defined taxonomy.
